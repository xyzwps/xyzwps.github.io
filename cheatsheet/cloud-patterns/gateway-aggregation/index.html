Gateway Aggregation pattern

提示：本文翻译自 Gateway Aggregation pattern - Azure Architecture Center。本文不保证翻译质量，并可能有删改。若你自觉英语尚可，强烈建议尝试看一下原文。

使用网关将多个单独的请求聚合成一个请求。当客户端必须多次调用不同的后端系统才能执行操作时，此模式很有用。

问题上下文

要执行单个任务，客户端可能必须对各种后端服务进行多次调用。 依赖许多服务来执行任务的应用程序必须在每个请求上消耗资源。 当任何新功能或服务被添加到应用程序时，都需要额外的请求，进一步增加资源需求和网络调用。 客户端和后端之间的这种闲聊会对应用程序的性能和规模产生不利影响。 微服务架构使这个问题变得更加普遍，因为围绕许多较小服务构建的应用程序自然会有更高数量的跨服务调用。

在下图中，客户端向每个服务发送请求（1,2,3）。每个服务处理请求并将响应发送回应用程序（4,5,6）。 在通常延迟较高的蜂窝网络上，以这种方式使用单个请求效率低下，可能导致连接中断或请求不完整。 虽然每个请求可以并行完成，但应用程序必须为每个请求发送、等待和处理数据，所有这些都在单独的连接上，增加了失败的机会。

解决方案

使用网关来减少客户端和服务之间的聊天。TODO: 这个翻译很奇怪 网关接收客户端请求，将请求分派到各种后端系统，然后聚合结果并将它们发送回请求客户端。

这种模式可以减少应用程序向后端服务发出的请求数量，并提高高延迟网络上的应用程序性能。

在下图中，应用程序向网关（1）发送一个请求。该请求包含一个附加请求包。 网关对这些进行分解，并通过将其发送到相关服务（2）来处理每个请求。 每个服务向网关（3）返回一个响应。网关将来自每个服务的响应组合起来，并将响应发送到应用程序（4）。 应用程序发出单个请求，仅从网关接收单个响应。

实现考量

  • 网关不应在后端服务之间引入服务耦合。

  • 网关应位于后端服务附近，以尽可能减少延迟。

  • 网关服务可能会引入单点故障。确保网关设计正确以满足应用程序的可用性要求。

  • 网关可能会引入瓶颈。确保网关具有足够的性能来处理负载，并且可以扩展以满足您的预期增长。

  • 对网关执行负载测试，以确保不会为服务引入级联故障。

  • 使用隔板、断路、重试和超时等技术实施弹性设计。

  • 如果一个或多个服务调用时间过长，超时并返回部分数据集可能是可以接受的。考虑您的应用程序将如何处理这种情况。

  • 使用异步I/O确保后端的延迟不会导致应用程序出现性能问题。

  • 使用相关ID实现分布式跟踪以跟踪每个单独的调用。

  • 监控请求指标和响应大小。

  • 考虑将缓存数据作为处理故障的故障转移策略返回。

  • 不要在网关中构建聚合，而是考虑在网关后面放置聚合服务。 请求聚合可能与网关中的其他服务具有不同的资源需求，并且可能会影响网关的路由和卸载功能。

    何时使用

    以下场景可以使用：

    • 客户端需要与多个后端服务通信才能执行操作。
    • 客户端可以使用具有显着延迟的网络，例如蜂窝网络。

    以下场景可能不适用：

    • 您希望减少客户端和单个服务之间跨多个操作的调用次数。在这种情况下，向服务添加批处理操作可能更好。
    • 客户端或应用程序位于后端服务附近，延迟不是一个重要因素。
